# Honeycomb — 설치 절차

대상: 현재 작업 중인 프로젝트 루트. 템플릿 출처: 이 스킬의 `files/`.

## 1. 기존 설치 확인 (덮어쓰기 금지)
- 대상에 이미 `docs/conventions.md` 또는 `docs/roles/`가 있으면 중복 설치다. 사용자에게 알리고 중단한다.

## 2. 템플릿 복사
- `files/` 트리를 프로젝트 루트로 복사한다 (`CLAUDE.md`, `docs/**`).
- **CLAUDE.md 충돌 처리:**
  - 대상에 `CLAUDE.md`가 없으면 → `files/CLAUDE.md` 그대로 복사.
  - 이미 있으면 → **덮어쓰지 말 것.** `files/CLAUDE.md`의 섹션(개발 원칙 / 프로젝트 맥락 / 문서 관리 / 역할 부여)을 기존 파일에 **병합**한다. 중복·충돌 섹션은 사용자에게 확인.
- `index.md`·`overview.md`·`shared.md`는 **빈 템플릿 그대로** 둔다 (프로젝트가 채운다).

## 3. overview 채우기 (사용자에게 질문)
- `docs/overview.md`가 `(미작성)` 상태이면 사용자에게 물어 채운다:
  - **프로젝트 개요**(업무/도메인): 무엇을, 왜 만드는가
  - **기술 컨텍스트**: 스택, 빌드/테스트/실행 명령, 코드 배치 규칙
- 업무 개요는 단위 작업 에이전트가 보통 안 읽고, 기술 컨텍스트는 implementer/verifier가 필요 시 읽는다.

## 4. 워크플로우 안내
설치 후 사용자에게 알린다:
- **feature 작성:** `docs/features/_template.md` 복사 → 완성 기준 작성 → 승인 게이트(`docs/workflow.md`) 통과 → `docs/index.md` 등록(상태 `대기`)
- **구현:** 별도 세션(또는 서브에이전트)에서 `"implementer 역할로 [기능] 구현해줘"`
- **검증:** 또 다른 별도 세션(또는 서브에이전트)에서 `"verifier 역할로 [기능] 검증해줘"`
- **핵심 원칙:** 구현자는 자기 결과를 검증하지 않는다. 검증자는 코드를 고치지 않는다. fresh·최소 컨텍스트.

## 5. (선택) 검증 강화
- 판정이 갈릴 수 있는 작업은 verifier를 여러 독립 에이전트로 돌려 교차 검증(3돌이). 결정론적·단순 기능은 1명으로 충분.

## 주의 — 템플릿 조정 포인트
- `files/docs/conventions.md` §6의 적용 예는 API 계약(`docs/api/`)을 예시로 든다. 대상 프로젝트 도메인에 맞게 예시만 바꿔도 된다(규칙 본문은 범용).

## 6. (참고) 기존 코드베이스 + 리팩토링 도입 시

운영 코드가 살아있는 프로젝트에 honeycomb을 깔고 *코드 리팩토링*까지 가는 경우의 검증된 패턴:

1. **문서 모듈화 ≠ 코드 모듈화.** 하네스 설치 + docs 3층 구조 ≠ 코드도 모듈화됨. 코드는 여전히 기존 layered/scattered 구조다. 별도 feature로 분할해야 한다.
2. **테스트 부재 코드베이스의 첫 feature**: `<topic>-char-test` (characterization test) — *현재 동작을 동결*만 한다. 운영 코드 0줄 수정. 다음 단계(VSA extract 등)의 안전망.
3. **운영 코드 무수정 강제**: feature 완성 기준에 `git diff <trunk>..HEAD -- <prod-path>/ | wc -l == 0` 항목 박기. char-test 본질 보존.
4. **만만한 단위부터** — 외부 HTTP·복잡 의존 없는 작은 모듈 1개로 패턴 검증. 큰 모듈은 패턴 확립 후. *Discovery 인벤토리*로 객관 점수화 (추측 금지).
5. **이전 시도 부검** — 같은 프로젝트에 이전 하네스/리팩토링 시도 흔적이 있으면 *왜 정체됐는지* 부검 먼저. 같은 함정 회피.

## 7. (참고) 신규(그린필드) 프로젝트 첫 feature

운영 코드가 아직 없는 신규 프로젝트는 첫 feature를 *제품 기능이 아니라 검증 차량*으로 삼는다. §6의 char-test와 동일 철학 — **첫 feature는 제품 가치가 아니라 하네스/안전망 확보가 목적**이다.

1. **외부 의존성 0 강제**: DB·큐·외부 API 전부 배제한 최소 기능 1개(예: health check 수준). feature 제약 조건에 "외부 의존성 추가 금지" 박기. 하네스가 도는지만 보고 싶은데 의존성이 끼면 그것 디버깅에 시간이 새어나간다.
2. **첫 구현 = 골격 부트스트랩**: 이 한 feature 구현이 앱 골격·빌드·컨테이너·테스트 러너를 통째로 만들게 한다. "하네스 1바퀴"와 "프로젝트 스캐폴딩"을 동시에 처리.
3. **완성 기준은 구현 없이도 PASS/FAIL 판정 가능하게**: 응답 본문·상태코드·미정의 경로 처리까지 못박아 verifier의 주관 판단 여지를 없앤다. 이게 *완성 기준 불변*이 작동하기 위한 전제다.
4. **단순·결정론적이라도 검증자 여럿(3돌이)을 돌린다**: §5는 "단순 기능은 1명으로 충분"이라 했지만, 첫 feature는 *검증 대상이 기능이 아니라 하네스(3돌이 머신) 자체*다. 차량인 동안은 풀 검증 루프를 돌려 머신을 검증한다.
