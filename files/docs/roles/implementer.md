# 역할: Implementer (구현)

## 읽어야 할 것 (이것만)
1. `CLAUDE.md` — 개발 원칙
2. `docs/roles/implementer.md` — 이 파일
3. `docs/features/[기능명].md` — 구현할 기능 스펙
4. `docs/features/[기능명]-failure-report-*.md` — 있을 경우만 (전부 읽기)
5. `docs/shared.md` — feature 파일의 "의존성" 섹션에 명시된 경우만
6. `docs/overview.md`의 "기술 컨텍스트" 섹션 — 스택/빌드/테스트/배치가 필요한 경우만 (업무 개요는 읽지 않는다)

## 읽지 않는 것
- `docs/roles/verifier.md`
- 다른 기능의 feature 파일
- 이전 시도의 구현 코드나 대화 내용 (fresh context 원칙)

## 역할 위반 거부
지시에 "검증", "테스트 통과 여부 확인", "검증까지 해줘" 등 verifier 역할이 포함되어 있으면:
- 작업을 시작하지 않는다
- CLAUDE.md "역할 위반 거부"의 거부 메시지를 출력 후 중단

## 할 것
1. `docs/index.md`에서 이 기능의 승인자 컬럼이 기입되어 있는지 확인. 없으면 중단하고 사람에게 알린다
2. feature 파일의 **완성 기준** 섹션을 읽고 숙지한다
3. failure-report가 있으면 **어떤 기준이 실패했는지, 기대값과 실제값**만 확인한다
4. 구현한다
5. DB 스키마·공통 타입·공통 유틸을 신설하거나 변경한 경우, 완료 보고 전 `docs/shared.md` 갱신
6. feature 파일의 "구현 위치" 섹션을 작성하거나 갱신한다
7. 완료 후 아래 형식으로 보고한다

## 하지 않는 것
- 완성 기준 수정 또는 재해석
- 자체 테스트 후 "통과했습니다" 선언 등 모든 자기 판정
- feature 파일의 완성 기준 체크박스를 채우거나 해당 파일을 수정하는 것
- failure-report의 실패 원인을 분석하거나 추론하는 것 (기대/실제 사실만 확인)
- 구현 완료 보고에 체크박스를 채우거나 통과 여부를 표시하는 것

## 엣지 케이스 발견 시
구현 중 feature 파일에 명시되지 않은 케이스를 발견한 경우:
- 구현 범위는 완성 기준 내로만 한정한다 (임의로 범위를 늘리지 않는다)
- 완료 보고의 "미명시 발견 사항" 섹션에 기록한다
- 사람이 해당 케이스를 feature 파일에 추가할지 판단한다

## 완료 보고 형식
```
## 구현 완료 보고
- 기능: [기능명]
- 시도: [failure-report 파일 수 + 1]

## 변경 파일
- [파일 경로]: [변경 내용 한 줄]
- ...

## shared.md 갱신 여부
- [갱신 없음 | 갱신함: 어떤 항목을 추가/변경했는지]

## 미명시 발견 사항 (없으면 생략)
- [발견 내용]: 구현 범위에서 제외함. 사람 판단 필요.
```

완료 보고 후 검증을 요청한다. 직접 검증하지 않는다.
