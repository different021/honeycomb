# 문서 관리 규칙

문서를 작성·수정하는 사람과 에이전트가 따르는 규칙. 새 문서 묶음을 만들 때 이 구조를 적용한다.

## 1. 3층 구조 — 인덱스 + 공유핵심 + 단위별 파일

모든 문서 묶음은 세 층으로 구성한다.

| 층 | 파일 | 역할 |
|----|------|------|
| 인덱스 | `index.md` | 단위 목록 + 한 줄 설명 + 상태. 스캔용 지도. **내용 금지.** |
| 공유핵심 | `_core.md` (+ 필요 시 `_payload.md` 등 복수) | **여러 단위가 공유하는 것만.** '모든 단위 공통'과 '일부 그룹만 공통'은 다른 파일로 나눈다 (↓ 다중 공유핵심). 한 단위만 쓰는 것은 그 단위 파일에. |
| 배경 | `_internal.md` (선택) | 계약이 아닌 배경(보류 항목, 내부 구현 메모, 설계 근거, 출처 등). 작업 시 거의 안 읽음. |
| 단위 | `<단위>.md` | 단위(API 1개, 기능 1개)별 자족 파일. |

읽는 사람/에이전트는 **공유핵심 + 필요한 단위 파일만** 읽는다. 묶음 전체를 읽지 않는다.
(근거: 글로벌 context-separation, CLAUDE.md "최소 컨텍스트" 원칙)

### _core 비대화 금지 (리트머스 테스트)
- **리트머스:** 어떤 단위 작업자가 `_core`를 읽을 때 **자기 작업과 무관한 섹션**이 있으면, 그 섹션은 `_core`에 잘못 들어간 것이다.
  - 예: ingest 전용 에러 규약, schema 전용 데이터 딕셔너리 본문 → `_core`가 아니라 해당 단위 파일로 내린다.
- **다중 공유핵심 — 리트머스가 "여러 단위 공유"보다 우선한다.** 일부 단위 그룹만 공유하고 나머지 단위엔 무관한 것은, 설령 2개 이상이 공유해도 `_core`에서 빼서 **별도 공유핵심 파일**로 분리한다.
  - 예: telemetry/event 페이로드 정의는 ingest·mqtt·telemetry-query·events-query만 공유 → `_payload.md`로 분리. `_core`에는 사실상 모든 단위가 읽어야 하는 것(봉투·진화 규칙·인증 등)만 남긴다.
- `_core`가 커져 "또 하나의 단일 문서"가 되면 분할의 목적(최소 컨텍스트)이 무력화된다. 단위 작업 1건의 읽기량(공유핵심 + 단위 1개)이 분할 전 단일 문서와 비슷하면 실패다.
- **계약 ≠ 배경:** 보류 항목·내부 구현 메모·설계 근거·배경 맥락은 공유핵심이 아니라 `_internal.md`(또는 단위 파일)에 둔다.

## 2. 분할 기준

- **단위 = 독립적으로 구현/검증 가능한 최소 덩어리.** (예: API 엔드포인트 그룹 1개, 기능 1개)
- 단위 하나 = 파일 하나. 한 파일은 그 단위만 자족적으로 설명한다.
- 단위 파일은 곧 implement/verify 사이클의 작업 단위가 된다.

## 3. 중복 금지 (drift 방지)

- **같은 사실을 두 곳에 "정의"하지 않는다.** 한 곳(정본)에 두고 나머지는 링크한다.
- **복붙뿐 아니라 재서술/복제도 금지.** 단위 파일은 규약을 다시 풀어쓰지(enumeration) 말고 링크만 한다. 같은 항목 목록이 두 파일에 나타나면 위반이다. 허용되는 것은 "이 단위가 무엇을 하는가" 한 줄 요약 + 정본 링크뿐.
- 공유되는 것(공통 봉투/타입, 규칙, enum·단위 정의)은 단위 파일에 복붙하지 말고 공유핵심에 두고 참조한다.
- 두 단위가 동일 사실을 공유하면, 한쪽에 본문을 두고 다른 쪽이 링크하는 대신 **둘 다 `_core`(또는 공통 정본)를 가리키게** 한다. 단위끼리 직접 상호 참조(A↔B)는 결합도를 만든다.
- **AI용 문서와 사람용 문서를 따로 손으로 관리하지 않는다.** 단일 출처에서 뷰를 생성한다.

## 4. 기계 SSOT 분리

- 계약의 기계가독 원본(OpenAPI/JSON Schema 등)이 있으면 별도 단일 SSOT로 둔다.
- 마크다운 문서는 그 위의 "맥락 + 탐색" 층이다. 기계 SSOT의 내용을 중복 정의하지 않는다.
- 사람 뷰(Swagger UI 등)와 AI 뷰(원본 json)는 같은 SSOT에서 나온다.

## 5. index 작성 규칙

- 한 항목 = 한 줄: `[제목](파일) — 한 줄 설명` (+ 상태가 있으면 함께).
- 내용을 index에 쓰지 않는다. 항상 파일로 분리하고 링크만 둔다.

## 6. 적용 예 — API 계약 문서

> 아래는 권장 구조 **예시**다(현재 `docs/api/`는 아직 존재하지 않음 — API 설계 확정 시 생성).

```
docs/api/
├── index.md            API 목록 + 한 줄 설명 + 상태 (내용 금지)
├── _core.md            2+ 단위 공유분만: 봉투(v, device_id, ts/recv_ts),
│                       데이터 모델, 진화 규칙, 인증/테넌시
├── _internal.md        계약 아님: 보류 항목, 내부 구현 메모, 출처
├── ingest.md           POST /ingest/{id}/... + 207 에러 규약(이 단위 전용)
├── telemetry-query.md  GET .../telemetry, /latest
├── events.md           events 수집 + 조회
├── device-mgmt.md      장비 등록/수정/삭제/회전
└── schema.md           /schema, /openapi.json + 데이터 딕셔너리(이 단위 전용)
```

- 단위 전용 규약(ingest의 207, schema의 데이터 딕셔너리 본문)은 `_core`가 아니라 **해당 단위 파일**에 둔다.
- 기계 SSOT(`openapi.json`)는 별도. 위 마크다운은 맥락+탐색 층.
